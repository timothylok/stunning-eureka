What it is

The AI Tools Evaluator scores a target (an AI tool description, or a GitHub repository) against a rubric using any LLM — local models through Ollama or cloud models through their APIs. Because the evaluation logic never depends on a specific vendor, the same target and rubric can be run through several models and compared side by side.

target + rubric → prompt builder → model provider → JSON validation → report (JSON + Markdown)

1 · Setup

  1. Install Node.js 22+ (the project uses built-in fetch and .env loading — there are no runtime dependencies).
  2. Run npm install once (dev tooling only: TypeScript and tsx).
  3. For local models, install Ollama and pull at least one model, e.g. ollama pull llama3.1:8b.
  4. For cloud models, put API keys in a .env file at the project root (see §10). .env is gitignored.

2 · Run an evaluation (CLI)

npm run evaluate -- --target targets/example.json --rubric rubrics/default.json --model llama3.1

All flags are optional:

FlagDefaultMeaning
--targettargets/example.jsonTarget file to evaluate
--rubricrubrics/default.jsonRubric to score against
--modelthe default entry in models.jsonModel id from models.json

The run prints the overall verdict and per-criterion scores, then saves a JSON + Markdown report pair to reports/.

3 · Evaluate a GitHub repo

The collector turns any public GitHub repository into a target file:

npm run collect -- --repo https://github.com/owner/name

It shallow-clones the repo into a temporary directory, distills it into a compact evidence bundle (≤ 20,000 characters, so 7B/8B local models can evaluate it too), writes targets/repo-<owner>-<name>.json, and deletes the clone. The evidence includes:

  • Repo stats — file count, language mix, and whether tests, CI workflows, a LICENSE, a Dockerfile, and a lockfile exist
  • File tree (up to 150 entries)
  • README excerpt
  • Manifests and configs (package.json, pyproject.toml, requirements.txt, Dockerfile, CI yaml)
  • Key source files (entry points, then largest shallow files)
  • Security scan — static indicators across seven categories: install hooks, obfuscation, network activity, dangerous file operations, cron/scheduled tasks, Dockerfile risks, and the declared dependency list. Pair with rubrics/security.json for a malware-risk evaluation. Static analysis only: no code execution and no CVE database lookups.

Then evaluate it with the repo rubric:

npm run collect  -- --repo https://github.com/owner/name
npm run evaluate -- --target targets/repo-owner-name.json --rubric rubrics/repo-quality.json --model agnes

4 · Compare models

npm run compare -- --target "owner/name"

Builds a Markdown table of the latest run per model for one target: every criterion score, the overall verdict, latency, and token usage. It prints to the terminal and saves reports/compare-<target>.md.

  • --target can be omitted when only one target has reports.
  • At least two models must have reports for the target.

5 · Web UI

npm run web

Serves the UI at http://localhost:3000 (change the port with the PORT environment variable).

  • Left panel — pick a model, target, and rubric, then Run evaluation. Disabled models appear greyed out. The dropdowns list whatever files exist in targets/ and rubrics/, so new files appear on reload.
  • Latest run — the verdict, per-criterion box-meter scores, and the model's strengths, weaknesses, recommendations, and risk flags.
  • Model comparison — the latest run per model for the most recently evaluated target, side by side.

The API behind it: GET /api/config, GET /api/reports, and POST /api/evaluate with {"model", "target", "rubric"} (filenames, not paths).

6 · Models & providers

models.json is the model registry. Each entry maps an id to a provider config:

{
  "default": "llama3.1",
  "models": {
    "llama3.1": { "type": "local",     "model_name": "llama3.1:8b",     "temperature": 0.2, "enabled": true },
    "agnes":    { "type": "openai",    "model_name": "agnes-2.0-flash", "base_url": "https://apihub.agnes-ai.com/v1",
                  "api_key_env": "AGNES_API_KEY", "temperature": 0.2, "enabled": true },
    "claude":   { "type": "anthropic", "model_name": "claude-opus-4-8", "enabled": false }
  }
}
FieldMeaning
typelocal (Ollama) · anthropic · openai (any OpenAI-compatible host) · google (not wired up yet)
model_nameThe provider's model id (e.g. llama3.1:8b, claude-opus-4-8)
enabledDisabled models are rejected by the CLI and greyed out in the UI
temperatureOptional. Used by local and openai providers; the anthropic provider never sends it (current Claude models reject sampling parameters)
base_urlopenai type only — point at any OpenAI-compatible host (default https://api.openai.com/v1)
api_key_envopenai type only — which environment variable holds the key (default OPENAI_API_KEY)

To enable a cloud model: put its key in .env, then flip "enabled": true in models.json. To add another OpenAI-compatible service, copy the agnes entry and change base_url, model_name, and api_key_env.

7 · Targets

A target is a JSON file in targets/:

{
  "name": "Tool Name",
  "description": "One paragraph on what it is.",
  "evidence": [
    "Observed fact one.",
    "Observed fact two."
  ]
}

Write them by hand for products and services, or generate them with the collector (§3) for GitHub repos. The model is instructed to judge only from the description and evidence, so richer evidence produces better-grounded scores.

8 · Rubrics

A rubric is a JSON file in rubrics/ defining the score scale and criteria:

{
  "name": "repo-quality-rubric",
  "scale": { "min": 1, "max": 10 },
  "criteria": [
    { "id": "correctness", "weight": 0.2, "description": "Does the code do what the project claims" }
  ]
}
  • Any scale works — the bundled rubrics use 1–5 (default.json, for AI tools) and 1–10 (repo-quality.json for repo quality; security.json for malware/security risk, where 10 = safest). Scores outside the scale are rejected.
  • weight is optional. When every criterion has one, the Markdown report adds a weighted composite (Σ score × weight).
  • Descriptions are shown to the model verbatim — write them as judging instructions.

9 · Reports

Every run writes two files to reports/ (gitignored):

  • <target>-<model>-<timestamp>.json — the full run record: scores, verdict, strengths/weaknesses/recommendations/risk flags, model id and name, rubric and scale, latency, and token usage. This is what the comparison views read.
  • …​.md — the same content as a readable Markdown report, including the weighted composite when the rubric has weights.
  • compare-<target>.md — written by npm run compare.

11 · Troubleshooting

SymptomFix
Ollama request failed / fetch failedStart Ollama and confirm the model is pulled: ollama list
Model "x" is disabled in models.jsonSet "enabled": true for that entry (and add its API key first)
…requires the XXX_API_KEY environment variableAdd the key to .env in the project root
Unknown model "x"Use an id listed in models.json — the error message lists them
Score for "x" is missing or outside min–maxThe model returned malformed output — re-run; small local models occasionally miss a criterion
Repository not found (collector)Check the URL; private repos are not supported by the collector
Need reports from at least 2 models (compare)Run the same target through a second model first

Local 7B/8B models are slower (40–90 s) and less consistent than cloud models. When a local model's scores look off, run the same target through a second model and compare — disagreement is signal.
